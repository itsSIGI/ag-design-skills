# Patch Mode — 局部修改协议

> 用户对已有产出说"改一下 X"、"颜色换成 Y"、"这个间距太大"时，走本协议。不需要重走完整 recipe 流程。

## 触发条件（满足全部才进入 Patch Mode）

1. 已有一份**本轮会话中** craft 产出的代码
2. 用户请求是**局部修改**（不涉及新增模块、删除模块、或改变页面结构）
3. 修改范围可明确圈定到具体的 slot / 组件 / 样式属性

**不符合**以上条件 → 回到标准流程（Step 1 起），不走 Patch Mode。

## Patch 分级

| 级别 | 范围 | 示例 | 操作 |
|------|------|------|------|
| **P1 纯样式调整** | 改 token 值、换颜色、调间距 | "间距再小点"、"换成灰色背景" | 直接改代码 → Step 3.6 Token 校验 |
| **P2 组件属性调整** | 改 variant、size、状态 | "按钮改成 secondary"、"卡片用 inset 样式" | 确认组件/variant 存在 → 改代码 → Step 3.6 |
| **P3 局部结构调整** | 调列宽、换排列方向、增减某字段 | "两列改三列"、"加个时间戳字段" | 更新 recipe 对应行 → 改代码 → Step 3.5 + 3.6 |
| **P4 组件替换** | 用另一个组件替代当前组件 | "表格换成卡片列表"、"Donut 改 Bar" | **回标准流程** Step 1 更新 recipe → 用户确认 → Step 2-4 |

## 执行流程

```
用户: "改一下 X"
     │
     ↓ 判定 Patch 级别
     │
     ├─ P1/P2: 直接改 → Token 校验（3.6）→ 输出
     │
     ├─ P3: 更新 recipe 对应行（无需用户确认）→ 改代码 → 质量自检（3.5）+ Token 校验（3.6）→ 输出
     │
     └─ P4: 回标准流程（告知用户"此修改涉及组件替换，需要更新 recipe"）
```

## 铁律（Patch Mode 同样适用）

- Token 校验（Step 3.6）**不可跳过**——即使只改了一个颜色值
- 组件变更必须在 `components-v2.md` 确认存在
- 修改后仍然输出 Decision Summary（标注 `Patch P1/P2/P3`）
- **不在 Patch 中偷加用户没要求的改动**——只改用户指出的部分

## 累积修改阈值

同一轮会话中 Patch 累计 ≥ 5 次，或 P3 累计 ≥ 3 次 → 建议用户"当前改动较多，建议重新走一次完整 recipe 确保整体一致性"。建议但不强制。
