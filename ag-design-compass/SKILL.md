# Design Compass — 需求拆解与设计策略推导

## 核心原则

> **没有文档来源的结论不输出。猜测不如追问。**

1. **证据优先**：所有角色痛点、行为描述必须引用原文；无法引用来源的结论不输出
2. **缺口显式化**：发现信息不足时，明确说"缺少 X 类型资料，请提供 Y"，不用推断填充
3. **Context 复用**：每次分析前加载 references/examples/ 中的已有案例，识别可复用的角色模式和流程结构
4. **角色视角贯穿**：每个设计策略必须能回答"为哪个角色解决了什么问题"
5. **奥卡姆剃刀**：以"移除此字段/功能后，角色还能完成核心任务吗？"为唯一保留标准。答案为"能"则删除
6. **结构化输出**：每轮分析的最终产出应结构完整（角色卡片 + 操作动线 + 设计策略 + 模块规格表），便于直接用于后续设计或开发

## Edge Cases（优先读，所有 Step 适用）

- **文档不足**：若关键信息缺失，在当前 Step 暂停并请求补充，不进入下一 Step
- **角色信息 < 2 个来源**：该角色痛点标注 `[待验证]`，不列入优先设计策略
- **跨角色矛盾**：显式标注冲突，不自行选边，让用户决策
- **文档类型未知**：先向用户确认"这是用户调研还是 PRD？"再摄入
- **业务上下文不足**：主动向用户索要文档链接或对应段落，不要用不充分的信息硬推

## Step 1: 加载已有 Context

**输入**：当前任务描述

**操作**：

1. **扫描示例**：列出 [references/examples/](references/examples/) 目录下的文件，选择与当前任务最相似的业务场景阅读
2. **加载方法论**：[references/design-goal-strategy-framework.md](references/design-goal-strategy-framework.md) 是 Step 4/5 的强制方法论，必须加载
3. **业务上下文索引**：查看 [references/domain-context.md](references/domain-context.md)，了解已有的业务领域知识索引
4. 向用户说明：已加载哪些方法论 + 哪些信息仍需补充
5. 如项目目录下存在 `.claude/design-log.md`，读取已有页面和设计约定，注入后续分析上下文

**跨 Skill 协作**：本 skill 的 Step 5.5 输出是 `ag-design-craft` 的直接输入——格式约定见 [`ag-design-craft/references/cross-skill-protocol.md`](../ag-design-craft/references/cross-skill-protocol.md)

**输出**：已知上下文摘要 + 缺口列表

---

## Step 2: 文档摄入

**输入**：用户提供的文档链接 / 文本内容 / 截图

**操作**：
1. 读取所有文档（飞书链接、截图、本地文件等）
2. 提取以下结构化信息，**每条标注来源段落**：
   - 背景与业务目标
   - 用户/客户反馈与引语（直接引用）
   - 业务流程描述
   - 已知痛点与瓶颈
   - 已有方案或限制条件
3. 标记文档类型：用户调研 / 会议记录 / PRD / 竞品分析 / 运营数据

**缺口处理**：若明显缺少某类资料，在此 Step 暂停询问：
> "目前缺少用户直接反馈的资料。是否有调研报告或用户访谈记录可以提供？"

**输出**：结构化文档摘要（带来源标注）

---

## Step 2.5: 竞品与行业参照分析（可选）

**输入**：Step 2 的文档摘要 + 用户提供的竞品信息（截图/链接/描述）

**触发条件**：以下任一即触发——
- 用户主动提供竞品截图或链接
- 文档中提及竞品或行业标杆
- 用户说"看看别人怎么做的"、"竞品分析"

**操作**：
1. 识别竞品的**关键差异化设计决策**（不是列功能清单，而是分析"他们用什么策略解决了什么问题"）
2. 对每个差异化决策标注：
   - **可借鉴**：适用于 AG 的场景 + 需要适配的点
   - **不适用**：原因（平台差异 / 用户群不同 / 技术约束）
   - **已有更优解**：AG 现有方案比竞品好的地方
3. 参考 [references/domain-context.md](references/domain-context.md) 中的行业上下文

**不做的事**：
- ❌ 不做功能对比矩阵（那是 PM 的活）
- ❌ 不评价"谁更好看"（用 arbiter 的维度评判）

**输出**：竞品设计策略分析（差异化决策 + 可借鉴/不适用/已有更优解）

---

## Step 2.8: 定量数据分析（可选）

**输入**：用户提供的埋点数据 / 漏斗分析 / A/B 实验结果 / 使用率报表

**触发条件**：用户提供了数据截图、CSV、或定量分析结果

**操作**：
1. 从数据中提取**设计相关的信号**：
   - **漏斗断点**：哪个步骤流失最多 → 对应 Step 3 的哪个角色/触点
   - **高频操作**：使用率最高的功能 → 优化优先级信号
   - **低使用率**：几乎没人用的功能 → 候选删减项
   - **A/B 结果**：哪个变体赢了 → 验证或推翻设计假设
2. 每条数据信号标注**置信度**：
   - `高`：数据量充足、统计显著
   - `中`：数据量够但有混杂变量
   - `低`：数据量不足或采集方式存疑（标注 `[待验证]`）
3. 数据信号注入 Step 3 角色痛点和 Step 5 策略优先级排序

**输出**：设计信号清单（带置信度）+ 与角色/痛点的映射

---

## Step 3: 角色识别与操作动线

**输入**：Step 2 的文档摘要

**操作**：
1. 提取文档中所有明示/隐含角色
2. 对每个角色输出角色卡片（格式见 [references/output-template.md](references/output-template.md)）：
   - **角色背景**：工作场景、设备、技术熟练度
   - **当前行为**：他们现在实际怎么做
   - **核心诉求**：一句话"我希望能…"
   - **当前痛点**：必须引用原文；< 2 个来源则标注 `[待验证]`
3. 绘制操作动线：各角色在完整流程中的触点顺序

**追问参考**：参见 [references/role-question-bank.md](references/role-question-bank.md) 中对应角色类型的标准问题

**输出**：角色卡片 × N + 操作动线

---

## Step 4: 需求目标 → 设计目标

**输入**：Step 3 的角色分析

**强制方法论**：必须遵循 [references/design-goal-strategy-framework.md](references/design-goal-strategy-framework.md) 第二节。**不读不允许进入 Step 4**。

**操作**：

1. **业务诉求分析**（4 维度）：明确目标用户、核心价值、价值表现、实现策略
2. **用户诉求分析**：用场景模版"在 [环境] 下，[用户] 做 [事] 来满足 [需求]"
3. **平衡产出**：
   - **需求目标**（What）：用"解决 [角色] 在 [场景] 中 [痛点]"句式
   - **设计目标**（How）：默认只写 **1 条主设计目标**，推荐句式：
     `让 [角色] 在 [关键场景] 中能够 [获得/完成什么体验结果]，同时 [满足一个关键约束]`
4. **强制自检**：
   - [ ] 可衡量：有具体指标可验证
   - [ ] 合理性：付出努力可实现
   - [ ] 可执行：与现有技术、资源匹配
   - [ ] 时效性：按需明确时限
   - [ ] 不是把"行动"当"目标"（红牌词："优化 XXX"、"改造 XXX"）

**输出**：需求目标列表 + 设计目标（默认 1 条）+ 自检结果

---

## Step 5: 设计策略拆解

**输入**：Step 4 的设计目标

**核心心法**：
> **目标是方向，策略是战术，设计方案是行动。**
> 执行链路：设计目标 → 设计策略 → **关键因素** → 设计点 → 设计方案。
> 不允许从"目标"直接跳到"UI 方案"。

### 5.1 发散切入点（内部思考）

至少用一个框架生成候选切入点：

| 框架 | 适用场景 |
|---|---|
| **用户旅程** | 关注"何时介入" |
| **JTBD** | 用户雇佣功能完成什么 job |
| **FOGG** | 行为 = 动机 × 能力 × 触发 |
| **AIDA** | Attention → Interest → Desire → Action |
| **AARRR** | 获取 / 激活 / 留存 / 收入 / 推荐 |
| **跨域类比** | 其他平台如何解决类似问题 |

至少得到 3 个候选方向，再收敛。

### 5.2 归纳排序（内部思考）

合并同类、剔除无效，排序：契合度 × 影响力 × 可执行性。

### 5.3 策略详细化

策略必须是**机制级**：描述系统如何运作、规则如何建立。
- 不合格：`大按钮`、`卡片样式`、`减少点击`
- 合格：`状态自动流转机制`、`异常纠偏机制`、`实验参数校验机制`

每条策略必须包含：
- **策略描述**：机制/规则/系统行为
- **解决的问题**：对应 Step 3 哪个角色的哪条痛点
- **关键因素**：决定效果的变量
- **需验证的假设**
- **优先级**：MVP / 后续迭代
- **使用的框架 + 为什么选它**

### 5.4 自检

- [ ] **系统性**：策略之间有全局关联
- [ ] **多样性**：≥2 个策略来自不同框架/视角
- [ ] **精准性**：综合考虑产品/需求/用户/行为

**输出**：问题→策略映射表 + 自检结果 + 待验证假设清单

---

## Step 5.5: UI 模块规格表（design-craft 入参）

**输入**：Step 5 的设计策略 + Step 3 的角色

**操作**：
1. 将每条策略拆为 UI 模块
2. 对每个模块输出规格表（格式见 [references/output-template.md](references/output-template.md)）：
   - **平台**：Web / Mobile / Both
   - **使用角色**
   - **优先级**：MVP / 后续迭代
   - **信息字段**：候选字段逐一过剃刀
   - **交互元素**：按钮/输入/选择器 + 触发行为
   - **状态变体**
   - **推荐页面原型**：匹配 `ag-design-craft/references/page-archetypes/` 中的原型
   - **推荐框架**：React / Vue / 其他（根据项目情况）
   - **视觉创新**：是 / 否（默认"否"。Landing page、营销页、品牌页、需要审美创新的展示型页面标注"是"。标注"是"时触发 `ag-design-vision` skill，先产出 Vision Spec 再交给 craft 执行）
3. 每个模块末尾附：
   - 视觉创新 = 否 → **→ 交给 ag-design-craft**
   - 视觉创新 = 是 → **→ 先交给 ag-design-vision 产出 Vision Spec → 再交给 ag-design-craft**

**输出**：结构化 UI 模块规格表（可直接传递给 `ag-design-craft`）

---

## Step 6: 输出结论

**输入**：本次分析的完整结论

**操作**：
1. 将 Steps 3–5.5 的完整分析整理为结构化 Markdown
2. 在输出顶部附上风险清单（所有 `[待验证]` 结论 + 开放性决策）
3. 将完整结论输出给用户

**输出**：结构化分析报告

---

## 失败回退

| 情况 | 处理 |
|---|---|
| 用户拒绝提供更多文档 | 标注所有 `[待验证]` 结论，继续推进，最终输出顶部列风险清单 |
| 文档读取失败 | 请用户复制文档文本或提供截图 |
| 角色信息严重不足 | 不输出该角色卡片，在 Step 5 末尾列信息缺口 |

## 回环修正机制

> compass 流程不是纯线性的——后续发现可以倒回修正前序结论。

### 允许回环的场景

| 场景 | 触发 | 操作 |
|------|------|------|
| **用户测试后推翻角色假设** | 用户说"实际使用发现角色 X 的痛点不对" | 回到 Step 3 修正角色卡片 → 重推 Step 4-5 |
| **craft 实现中发现策略不可行** | craft 反馈"策略 Y 在当前设计系统无法实现" | 回到 Step 5 调整策略，标注技术约束 |
| **新增文档改变结论** | 用户追加了新的调研报告或数据 | 回到 Step 2 重新摄入 → 增量更新后续 Step |
| **stakeholder 否决设计目标** | 用户说"老板不同意这个方向" | 回到 Step 4 修改设计目标 → 重推 Step 5-5.5 |

### 回环规则

1. **标注修改原因**：每次回环在输出中标注 `[修正 v2: 原因]`
2. **增量更新**：只重做受影响的 Step，不从头重来
3. **保留历史**：修改前的结论不删除，标注 `[v1, 已被 v2 替代]`
4. **置信度升级**：被验证过的结论从 `[待验证]` 升级为 `[已验证: 来源]`
